# سياسة إصدارات الـ API (API Versioning Policy)

هذا المستند **توثيقي فقط**. لا يوجد أي تغيير في الـ routing الحالي — `/health` و`/health/database`
يفضلوا كما هم في هذا الـ Milestone. السياسة دي بتتطبّق بداية من أول Endpoint فعلي بيقدّم منطق
أعمال (Milestone قادم).

## المسار الأساسي

كل Endpoint بيقدّم منطق أعمال (مش Endpoints البنية التحتية زي فحوصات الحالة) هيكون تحت:

```
/api/v1/...
```

فحوصات الحالة (`/health`, `/health/database`) بتفضل خارج نطاق الإصدار (Unversioned) عمدًا، لأنها
بنية تحتية بتُستخدم من أدوات المراقبة (Load Balancers، Kubernetes probes...) اللي مش المفروض
تتأثر بتغييرات نسخة الـ API التجارية.

## قاعدة التغييرات الجذرية (Breaking Changes)

أي تغيير بيكسر التوافق مع عملاء موجودين بالفعل (تغيير شكل الاستجابة، حذف حقل، تغيير معنى حقل،
تغيير كود خطأ ثابت...) **يستوجب نسخة جديدة** (`/api/v2`)، ومينفعش يتعمل داخل `/api/v1` الحالية.

أمثلة على ما يُعتبر Breaking Change:
- حذف حقل من استجابة موجودة.
- تغيير نوع بيانات حقل (مثال: من `string` إلى `object`).
- تغيير كود خطأ ثابت (`code` في `docs/api-error-conventions.md`).
- تغيير سلوك افتراضي متوقَّع من عميل موجود.

أمثلة على ما **لا** يُعتبر Breaking Change (ويُسمح داخل نفس النسخة):
- إضافة حقل اختياري جديد لاستجابة موجودة.
- إضافة Endpoint جديد بالكامل.
- إضافة قيمة جديدة لكود خطأ (`code`) من غير حذف أو تغيير القيم الموجودة.

## قاعدة التوافق مع الإصدارات القديمة

النسخ الحالية (`/api/v1`) تفضل شغالة ومدعومة قدر الإمكان حتى بعد إصدار نسخة أحدث (`/api/v2`) —
لا يتم إيقاف أي نسخة قديمة فجأة. أي خطة لإيقاف دعم نسخة معينة لازم تُوثَّق هنا بفترة سماح واضحة
قبل التنفيذ الفعلي، وقت ما يحصل ده فعليًا مستقبلًا.
