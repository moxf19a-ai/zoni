# اتفاقيات أكواد الأخطاء (API Error Conventions)

هذا المستند **توثيقي فقط** — يوصف الشكل اللي هتتبناه كل استجابة خطأ في الـ API بداية من أول
Milestone بيقدّم منطق أعمال فعلي (Milestone 4 فما بعد). لا يوجد أي middleware أو معالج أخطاء
منفّذ في المشروع حتى الآن.

النوع المطابق لهذا الشكل معرّف بالفعل في الكود كتحضير مسبق:
`apps/api/src/presentation/http/api-response.type.ts` (`ApiErrorResponse`).

## شكل استجابة الخطأ الموحد

```json
{
  "success": false,
  "error": {
    "code": "AUTH_INVALID_CREDENTIALS",
    "message": "Invalid email or password.",
    "details": null,
    "requestId": "a1b2c3d4-5678-90ab-cdef-1234567890ab",
    "timestamp": "2026-07-06T10:00:00.000Z"
  }
}
```

### شرح كل حقل

| الحقل | إلزامي؟ | الوصف |
|---|---|---|
| `code` | نعم | كود مقروء آليًا (Machine-readable)، `UPPER_SNAKE_CASE`، بادئة توضح النطاق (مثال: `AUTH_*`, `VALIDATION_*`, `BILLING_*`). لا يتغير أبدًا بمرور الوقت — العملاء (Frontend، SDKs مستقبلية) بيعتمدوا عليه للتفرع البرمجي، مش على نص `message`. |
| `message` | نعم | رسالة مقروءة للبشر (Human-readable)، بالإنجليزية دائمًا على مستوى الـ API نفسه (الترجمة للعربية أو غيرها تتم في طبقة العرض/الفرونت إند، مش في الـ API). لا تُستخدم للتفرع البرمجي أبدًا. |
| `details` | اختياري | أي معلومات إضافية منظّمة (مثال: قائمة أخطاء التحقق من كل حقل في نموذج). `null` أو محذوف تمامًا لو مفيش تفاصيل إضافية. |
| `requestId` | نعم | معرّف فريد لكل طلب، بيُنشأ في بداية دورة حياة الطلب (middleware مستقبلي في M4)، ويُستخدم لربط شكوى العميل بسجلات الـ Logger المطابقة. |
| `timestamp` | نعم | توقيت حدوث الخطأ بصيغة ISO 8601 UTC. |

## قواعد تسمية الكود (`code`)

- `UPPER_SNAKE_CASE` دائمًا.
- بادئة تدل على النطاق/الموديول المسؤول: `AUTH_`, `VALIDATION_`, `BILLING_`, `INSTAGRAM_`, إلخ.
- الكود ثابت عبر كل نسخ الـ API — تغييره يُعتبر Breaking Change (راجع `docs/api-versioning-policy.md`).

## ما لا يدخل ضمن هذا الشكل

أخطاء الشبكة أو تعطل السيرفر الكامل (مثل 502/504 من الـ Load Balancer قبل ما تصل للتطبيق نفسه)
مش مسؤولية هذا التنسيق — التنسيق ده بيغطي فقط الأخطاء اللي التطبيق نفسه بيرجعها بوعي.
